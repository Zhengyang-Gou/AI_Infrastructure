# ZeRO
## ZeRO 解决问题的高层逻辑
### 痛点：算力跟得上，显存却爆了
随着 NLP 领域 Transformer 架构的爆发，模型体积正在狂飙突进，比如从 Bert-large (0.3B) 一路涨到 T5 (11B)。但尴尬的是，单张 GPU 的显存根本装不下这些庞然大物。

- 传统数据并行彻底失效：DP 的做法是在每张显卡上都完整复制一份模型，然后把数据切分。因为每张卡都要装完整的模型，导致 32GB 的显卡在面对大于 14 亿 (1.4B) 参数的模型时直接 OOM

### 现有的方法为什么不够好
为了解决显存塞不下的问题，业界想过很多办法（比如模型并行、流水线并行、CPU卸载），但都有致命缺陷，其中被寄予厚望的模型并行表现最典型：

- MP 的原理与缺陷：MP 是把模型垂直切分，将一层网络的计算和参数分到多张卡上。这虽然解决了显存问题，但每一层计算完都需要极其频繁的通信

- 跨节点：在同一个物理节点里，MP 勉强能用 。但一旦模型大到需要跨节点训练，通信效率就会雪崩。作者亲测用 MP 跨两个节点训练 40B 模型时，GPU 的算力利用率连硬件峰值的 5% 都不到

### 内存占用
内存占用归为两大类：

- 模型状态：这是内存消耗的大头，包括优化器状态（比如 Adam 里的动量和方差）、梯度和参数

- 剩余状态：剩下的内存被激活值、临时缓冲区以及由于内存碎片化导致无法使用的空间占据

### ZeRO 解决方法
针对这两大类内存消耗，ZeRO 给出了对应的解决方案：

- 对付模型状态 -> ZeRO-DP：它保留了 DP 计算/通信效率高的优点，同时吸取了 MP 内存效率高的优点。不复制，只划分。通过三个渐进的阶段（划分优化器状态、划分梯度、划分参数），可以将这部分内存消耗最高减少 $N_d$ 倍

- 对付剩余状态 -> ZeRO-R：专门用来优化激活值、缓冲区大小，并主动管理内存防止碎片化

- $\Psi$：模型大小，图中的例子是 7.5B

图中的长条代表显存占用：蓝色代表参数，橙色代表梯度，绿色的大头代表优化器状态 

- Baseline: data parallel，每张卡都保存了完整模型状态，完全重复，加更多 DP 卡，只是算得更快，不会让每卡的 model-state 显存下降，普通 DP 对单卡显存没有帮助

- P_os：第一阶段只切 optimizer states，把优化器状态切成 N_d 份，每个 DP rank 只保存自己那 1 份；但参数和梯度仍然完整保留。每个 process 只更新自己那部分 optimizer states，并在 step 末尾 all-gather 更新后的参数。2Ψ + 2Ψ + K * Ψ / N_d，大约 4x memory reduction，同时通信量和普通 DP 相同

- P_os+g：第二阶段再切 gradients，既然每个 DP rank 只负责更新自己那部分参数，那它其实只需要那一部分参数对应的归约后梯度，所以不需要再让每张卡都持有完整梯度。不同参数块的梯度归到不同 process，归完之后就可以释放。2Ψ + (2 + K)Ψ / N_d，大约 8x reduction，并且通信量仍和 baseline DP 相同

- P_os+g+p：第三阶段连 parameters 也切掉，此时每张卡连参数都不再整份保存，只保存自己负责的那一片；forward/backward 需要别的参数分片时，再从相应 DP process 取过来。通信量提高到 baseline DP 的 1.5x，但显存会进一步按 N_d 线性下降。(2 + 2 + K)Ψ / N_d = 16Ψ / N_d，可以达到 64x 的 model-state memory reduction

## 内存都去哪了？
一个拥有 15 亿（1.5B）参数的 GPT-2 模型在 16 位精度下仅需要 3GB 的内存来存储其权重（或参数），然而，使用 Tensorflow 或 PyTorch 却无法在一个拥有 32GB 内存的单张 GPU 上训练它

在模型训练期间，大部分内存被模型状态所消耗，即由优化器状态、梯度和参数组成的张量。除了这些模型状态之外，剩余的内存则被激活值、临时缓冲区和不可用的碎片化内存所消耗，我们将它们统称为残余状态。

### 模型状态：优化器状态、梯度和参数
在训练过程中，大部分的设备内存被模型状态所消耗。以 Adam 为例，Adam 需要存储两个优化器状态来计算更新：时间平均动量，以及梯度的方差。因此，要使用 Adam 训练模型，必须有足够的内存来保存动量和梯度方差的副本。此外，还需要足够的内存来存储梯度和权重本身。在这三种与参数相关的张量中，优化器状态通常消耗的内存最多，尤其是在应用混合精度训练时

目前在当代 NVIDIA GPU 上训练大模型的最先进方法是混合精度（fp16/32）训练，其中参数和激活值以 fp16 格式存储，这使得在这些 GPU 上能够使用高吞吐量的 Tensor Core 单元。在混合精度训练期间，前向和反向传播都使用 fp16 的权重和激活值执行。然而，为了在反向传播结束时有效地计算和应用更新，混合精度优化器会保留一份 fp32 格式的参数副本，以及所有其他优化器状态的 fp32 副本

使用 Adam 对参数量为 $\Psi$ 的模型进行混合精度训练时，需要足够的内存来保存参数和梯度的 fp16 副本，它们的内存需求分别为 $2\Psi$ 字节。此外，它还需要保存优化器状态：参数、动量和方差的 fp32 副本，其内存需求分别为 $4\Psi$、$4\Psi$ 和 $4\Psi$ 字节。让我们用 $K$ 来表示优化器状态的内存乘数，也就是说，存储它们所需的额外内存为 $K\Psi$ 字节。混合精度的 Adam 其 $K=12$。总而言之，这将导致 $2\Psi + 2\Psi + K\Psi = 16\Psi$ 字节的内存需求

### 残余内存消耗
激活值在训练期间可能会占用大量内存。举个具体的例子，在序列长度为 1K、批次大小为 32 的情况下，训练 15 亿参数的 GPT-2 模型大约需要 60GB 的内存。激活值检查点是一种常用的方法，可以以增加 33% 重计算开销为代价，将激活值内存减少到大约总激活值的平方根。这会将该模型的激活值内存消耗降低到 8GB 左右。尽管缩减幅度很大，但对于更大的模型来说，即使使用了激活值检查点技术，激活值内存仍然可能变得非常庞大。例如，一个拥有 1000 亿参数的类 GPT 模型，即使在使用激活值检查点技术时，在批次大小为 32 的情况下也需要大约 60GB 的内存

临时缓冲区用于存储中间结果，在大型模型中也会消耗不可忽视的内存量。像梯度 All-Reduce 或梯度范数计算这样的操作，往往会在应用操作之前将所有梯度融合成一个单一的扁平缓冲区，以努力提高吞吐量。例如，跨设备的 All-Reduce 带宽会随着消息大小的增加而提高。虽然梯度本身通常作为 fp16 张量存储，但取决于具体的操作，融合后的缓冲区可能是一个 fp32 张量。当模型尺寸较大时，这些临时缓冲区的大小是不容忽视的。例如，对于一个 15 亿参数的模型，一个扁平的 fp32 缓冲区就需要 6GB 的内存

内存碎片化：此外，即使有充足的可用内存，也可能出现可用内存耗尽的情况。如果没有足够的连续内存来满足某个内存请求，该请求就会失败，即使总的可用内存大于请求的量。在训练超大模型时，我们观察到了显著的内存碎片化现象，在一些极端情况下，即使仍有超过 30% 的空闲可用内存，也会导致内存溢出问题

## ZeRO 洞察与概述
ZERO 包含两套优化方案：旨在减少模型状态内存占用的 ZeRO-DP，以及旨在减少残余内存消耗的 ZeRO-R。“效率”在这里是关键：如果没有效率这个约束条件，通过将所有参数状态移至 CPU 内存或任意增加模型并行（MP）的程度等简单的解决方案，也能达到减少内存占用的目的

### ZERO-DP
- DP 比 MP 具有更好的扩展效率，因为 MP 在降低计算粒度的同时增加了通信开销。超过某个特定点后，较低的计算粒度会降低每个 GPU 的效率，而增加的通信开销会阻碍跨 GPU 的可扩展性，尤其是在跨越节点边界时。相反，DP 既有较高的计算粒度又有较低的通信量，从而能实现高得多的效率

- DP 的内存效率低下，模型状态在所有数据并行进程中被冗余存储，相反，MP 通过对模型状态进行切分来获得内存效率

- DP 和 MP 都在整个训练过程中保留了所有需要的模型状态，但并非所有状态在所有时刻都是必需的，每一层对应的参数仅在该层的前向传播和反向传播期间才被需要

ZERO-DP 对模型状态进行切分而不是复制，并使用动态通信调度，该调度利用了模型状态内在的时间特性，同时将通信量降至最低。通过这样做，ZERO-DP 随着 DP 程度的增加线性地降低了模型在每台设备上的内存占用，同时将通信量保持在接近默认 DP 的水平，从而保留了计算效率

### ZERO-R
- MP 虽然对模型状态进行了切分，但往往需要复制激活内存。如果我们对线性层的参数进行垂直切分并在两个 GPU 上并行计算，那么每个 GPU 都需要完整的激活值来计算其对应的分区

- 对于像 GPT-2 或更大的模型，其算术强度非常大（$\ge10K$），并且随隐藏层维度呈线性增加；这使得隐藏激活检查点的数据移动成本成为可能，即使在带宽较低的情况下也是如此

ZERO 通过跨 GPU 切分激活检查点消除了 MP 中的内存冗余，并在需要时使用 allgather 操作来重构它们。激活内存的占用量与 MP 程度成正比减少。对于超大型模型，ZeRO 甚至可以选择将激活分区移动到 CPU 内存中，同时由于这些模型具有极大的算术强度，仍然能够取得良好的效率

ZERO 通过将激活检查点和梯度移动到预先分配好的连续内存缓冲区中，来执行动态内存碎片整理。这不仅增加了内存的可用性，还通过减少内存分配器寻找空闲连续内存所需的时间，从而提升了系统效率

## Deep Dive into ZeRO-DP
现有的 DP 方法在每个设备上复制模型状态，引入了显著的内存开销；而 ZeRO-DP 通过在数据并行进程中对优化器状态、梯度和参数进行划分，消除了这种内存冗余。我们将 ZeRO-DP 的分为三个优化阶段：$P_{os}$、$P_{g}$ 和 $P_{p}$

### $P_{os}$：优化器状态划分
对于 DP 度 N_d，把 optimizer states 分成 N_d 份，第 i 个 data parallel process 只更新第 i 份 optimizer states。于是，每个进程只需要存 1/N_d 的 optimizer states，并且只负责更新 1/N_d 的参数；在训练 step 结束时，再通过一次 all-gather，让所有 data parallel process 拿到完整更新后的 parameters

优化器状态划分后的内存消耗从 $4\Psi+K\Psi$ 减少到了 $4\Psi+\frac{K\Psi}{N_{d}}$

### $P_{g}$：梯度划分
由于每个数据并行进程仅更新其对应的参数分区，因此它只需要对应参数规约后的梯度。因此，在反向传播期间，当每一层的每一个梯度计算出并变得可用时，我们仅在负责更新对应参数的数据并行进程上对它们进行规约。规约完成之后，我们不再需要这些梯度，它们的内存可以被立刻释放。这将在保存梯度所需的内存占用从 $2\Psi$ 字节减少到了 $\frac{2\Psi}{N_{d}}$ 字节

是一个 Reduce-Scatter 操作，其中对应于不同参数的梯度被规约到了不同的进程中。为了在实践中提高效率，使用了一种分桶（bucketization）策略，即将对应于特定分区的所有梯度放入同一个桶中，并一次性对整个桶执行规约。在我们的方案中，我们在分区边界执行 reduction（规约）而不是 all-reduce，以减少内存占用并让计算和通信重叠

模型状态的内存需求从 $2\Psi+14\Psi=16\Psi$ 字节减少到了 $2\Psi+\frac{14\Psi}{N_{d}}\approx2\Psi$ 字节

### $P_{p}$：参数划分
每个进程现在仅存储与其分区对应的参数。当进行前向和反向传播需要其分区之外的参数时，它们会通过广播从相应的数据并行进程处接收。虽然乍一看这似乎会引起显著的通信开销，但这种方法仅将基准 DP 系统的总通信量增加到了 1.5 倍，同时实现了与 $N_{d}$ 成比例的显存减少

将一个大小为 $\Psi$ 的参数模型的内存消耗从 $16\Psi$ 减少到了 $\frac{16\Psi}{N_{d}}$，只要有足够数量的设备来共享模型状态，ZeRO 就能赋能 DP 容纳任意大小的模型

## Communication Analysis of ZeRO-DP
ZeRO-DP 在使用 $P_{os}$ 和 $P_{g}$ 时不会产生额外的通信量，同时能实现高达 8 倍的内存减少

当在使用 $P_{os}$ 和 $P_{g}$ 的基础上再使用 $P_{p}$ 时，ZeRO-DP 最多会产生 1.5 倍的通信量，同时能将内存占用进一步按 $N_{d}$ 倍减少

### 数据并行通信量
在数据并行训练期间，在计算下一步的更新之前，系统会在反向传播结束时平均所有数据并行进程中的梯度。这种平均操作是使用 All-Reduce 通信集合操作来完成的。最先进的 All-Reduce 实现采用两步法：第一步是 Reduce-Scatter 操作，该操作将数据的不同部分规约到不同的进程上。第二步是 All-Gather 操作，每个进程在此步骤中收集所有进程上已规约的数据。这两个步骤的结果就等同于一次 All-Reduce。Reduce-Scatter 和 All-Gather 都使用流水线方法实现，对于包含 $\Psi$ 个元素的数据，它们各自会导致 $\Psi$ 的数据移动量。因此，标准 DP 在每个训练步长中会产生 $2\Psi$ 的数据移动量

### ZeRO-DP 通信量
#### 开启 $P_{os+g}$ 时的通信量
在进行梯度划分后，每个进程只存储更新其对应参数分区所需的梯度部分。因此，ZeRO 不需要完整的 All-Reduce，而是只需要对梯度执行一次 Scatter-Reduce 操作，产生的通信量为 $\Psi$。在每个进程更新了它所负责的参数分区之后，会执行一次 All-Gather 操作，从所有数据并行进程中收集所有已更新的参数。这同样会产生 $\Psi$ 的通信量。所以，每个训练步长的总通信量为 $\Psi + \Psi = 2\Psi$，与基线 DP 完全相同

#### 开启 $P_{os+g+p}$ 时的通信量
在进行参数划分后，每个数据并行进程仅存储它所负责更新的参数。因此，在前向传播期间，它需要接收所有其他分区的参数。在对模型中对应于特定分区的结构计算前向传播之前，负责该分区的数据并行进程可以将对应的权重广播给所有的数据并行进程。一旦该分区的前向传播完成，这些参数就可以立刻被丢弃。此时的总通信量为 $\frac{\Psi \times N_{d}}{N_{d}} = \Psi$。换句话说，我们将参数的 All-Gather 操作重新进行了调度，将其分散到整个前向传播过程中，并在参数用完后立刻将其丢弃。但请注意，在反向传播阶段，这个 All-Gather 操作需要以相反的顺序再发生一次

总的通信量是这两次前向和反向中的 All-Gather 产生的通信量之和，再加上计算完梯度后的 Reduce-Scatter 所产生的通信量。总通信量因此为 $3\Psi$，是基线（$2\Psi$）的 1.5 倍

## Deep Dive into ZeRO-R
### $P_{a}$：激活值检查点划分
模型并行（MP）虽然把参数切开了，但 activation 往往仍然是复制的。比如一个线性层按列切到两张卡上，虽然权重被分片了，但每张卡还是需要拿到完整输入 activation，才能算自己那一部分输出

Pa 的做法是：
- forward 某层算完之后，不再把这层 checkpoint activation 以复制状态留在所有 MP ranks 上

- 而是把它按 MP ranks 分片存储

- 等到 backward 重算或者真正需要它时，再通过 all-gather 临时把完整 activation 拼回来

Pa 带来的 activation 显存下降与 MP 度成正比

如果模型特别大、设备显存特别紧，那么这些分片后的 activation checkpoints还可以继续 offload 到 CPU，这叫 Pa+cpu。它的效果是：

- activation memory 几乎为零

- 但要付出额外通信/数据搬运代价

### $C_{B}$：固定大小缓冲区
训练里很多操作为了效率，会把很多小 tensor fuse 到一个大 buffer 里再做。问题在于，如果总是把全模型相关 tensor 融成一个大 buffer，那么这个 buffer 的大小会跟模型规模一起线性长大。模型越大，buffer 越吓人

我们别让 buffer 无限跟着模型长，改用一个够大但固定大小的 buffer

### $M_{D}$：内存碎片整理
训练里的 memory fragmentation 主要来自两种交织：
- forward + checkpointing 阶段
    - 被 checkpoint 保留下来的 activation：长生命周期
    - 那些可重算、算完就扔的 activation：短生命周期

- backward 阶段
    - parameter gradients：长生命周期
    - activation gradients 和一些中间缓冲：短生命周期

这些短命和长命对象在内存里反复穿插分配/释放，就会把显存切得很碎，总空闲显存看起来还不少，但找不到一块足够连续的大空间

ZeRO 做的是 on-the-fly memory defragmentation，通过把 activation checkpoints 和 gradients 移到预分配的 contiguous memory buffers 里，不仅提升可用显存，还减少 allocator 找连续空间的时间
